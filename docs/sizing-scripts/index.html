Wiz Sizing Scripts

Organized copies of Wiz resource-discovery and sizing scripts for estimating billable units before connectors are enabled. The scripts live in the repository; gists are kept as instruction pages.

sizing resource discovery python powershell

Overview

The scripts estimate billable units before Wiz has live connector data. After a connector is enabled, the Wiz Settings > Licenses page is the better source of truth.

Code

Counts active developers across VCS and CI/CD systems using commits and activity from the last 90 days.

View Code scripts
Cloud

Counts compute, data, registry, and infrastructure resources across cloud and infrastructure providers.

View Cloud scripts
SaaS

Holds SaaS-specific sizing scripts, including the hardened Microsoft 365 sizing script.

View SaaS scripts
Privacy note: Some scripts create local output files with contributor or resource details. Review those files before sharing them.

Microsoft 365 Canonical Path

The Microsoft 365 sizing script is organized with the rest of the sizing scripts. Use this path for new runbooks and customer instructions.

Invoke-WebRequest -Uri https://raw.githubusercontent.com/adilio/wiz-tools/main/sizing-scripts/saas/microsoft-365/365_Sizing_Script.ps1 -OutFile 365_Sizing_Script.ps1
./365_Sizing_Script.ps1

The older published m365-sizing-xl URL is retained only as a compatibility documentation page for links that were already shared. It points to the same canonical script path above.

Common Flags

  • --all scans all accessible accounts, projects, or subscriptions where supported.
  • --data includes buckets and databases where supported.
  • --images includes registry container images where supported.

Modified Script Notes

Scripts marked modified keep the original sizing intent, but include local hardening for field-reported runtime or reliability issues. The complete inventory is tracked in sizing-scripts/SCRIPT_STATUS.md.

Microsoft 365 Sizing Script

AreaOriginal behaviorUpdated behaviorWhy it changed
Temporary app lifecycleUsed a fixed temporary app name and cleaned up on the successful pathCreates a unique temporary Entra ID app by default and removes it from a finally blockReduces risk of reusing or leaving behind the temporary app during failed runs
Cloud Shell sign-inUsed normal Microsoft Graph sign-inUses device-code authentication by defaultImproves reliability in Azure Cloud Shell
Long-running scansUsed one app-only access token before scanningTracks expiry and refreshes app-only Graph tokens before they expirePrevents large tenants from failing after an hour because the token expired
Graph callsRetried throttling and API errors, then could stop without a clear final failureHonors retry guidance and reports clear partial counts before cleanup if the scan failsPrevents incomplete counts from looking like successful final results
Output compatibilityPrinted the original FINAL COUNTS blockKeeps the same final block while adding elapsed progress and status linesPreserves existing runbooks while making long runs easier to monitor

Azure DevOps Active Developer Count

AreaOriginal behaviorUpdated behaviorWhy it changed
Repository loopAccumulated repositories across projects and then looped over the full accumulated list for each projectScans only the repositories returned for the current projectPrevents repeated rescans that can make large organizations look like they are running indefinitely
Project and commit retrievalRequested projects and commits without explicit operator-visible paging controlsAdds paged project retrieval and paged commit retrieval with --project-page-size and --commit-page-sizeImproves behavior in large organizations and repositories while making API work more visible
Runtime visibilityPrinted repository and developer discoveries but no elapsed progress cadenceAdds elapsed status lines and --progress-intervalHelps operators distinguish slow progress from a hung scan
GuardrailsNo built-in cap for repositories, commits, or wall-clock runtime during troubleshootingAdds --max-repositories, --max-commits-per-repo, and --max-run-minutesAllows safe pilots before committing to a full organization scan
Interruption and checkpointsCtrl-C exited without writing partial resultsWrites partial outputs before exit; optional checkpoint outputs are available with --checkpoint-intervalPreserves useful counts from long-running scans without changing normal-run output too much
API reliabilityAPI errors were logged or skipped without retry/backoff controlsAdds --max-retries, --retry-delay, and --fail-fastHandles transient Azure DevOps failures while preserving a strict troubleshooting mode
Privacy optionDeveloper email addresses were printed to the terminalKeeps that default for compatibility; --mask-emails opts into terminal maskingPreserves familiar output while adding a safer screen-share mode

GCP Resource Count

AreaOriginal behaviorUpdated behaviorWhy it changed
Data workload defaultData counters were enabled internally even when --data was not suppliedData buckets, PaaS databases, data warehouses, and non-OS disks now require --dataAligns runtime behavior with documented flags and shortens default discovery
Project scope--all scanned every accessible projectAdds --max-projects, --include-project-regex, and --exclude-project-regexLets operators pilot or exclude noisy project families such as Apps Script-associated projects
Resume behaviorLong scans had to restart from the beginning after interruptionAdds --start-after-project for sorted project listsSupports practical continuation after the last completed project
API guardrailsPaginated API calls could continue until the service finished or failedAdds --request-timeout, --max-pages-per-request, and --max-run-minutesBounds problematic calls and records partial data rather than silently running for many hours
Partial outputCSV output was written after the full scanWrites partial CSV output on interruption, bounded-run exit, and optional --checkpoint-interval checkpointsPreserves useful inventory from long scans that crash or are stopped without changing normal-run output too much
Alternative collectionNo explicit fallback path for customers unable to run a full scanAdds --inventory-instructions with Cloud Asset Inventory guidanceGives customers a lower-friction way to provide counts when API scanning is too expensive
